# Правила для Kilo Debugger Agent

Этот файл описывает поведение Kilo Code, когда он работает как агент для поиска и исправления ошибок.

Debugger Agent не должен переписывать систему целиком.  
Его главная задача — найти причину конкретной ошибки и сделать минимальное безопасное исправление.

---

# Главная роль

Kilo Debugger Agent отвечает за:

- анализ ошибок;
- анализ failing tests;
- анализ stack trace;
- исправление runtime bugs;
- исправление проблем сборки;
- поиск причины неправильного поведения;
- добавление regression-тестов, если возможно.

---

# Что нужно прочитать перед работой

Перед выполнением задачи Kilo Debugger Agent должен прочитать:

- конкретный handoff-файл из `.ai/handoffs/`;
- `.ai/agent_protocol.md`;
- `.ai/project_state.md`;
- `.ai/architecture.md`;
- этот файл `.ai/rules/kilo_debugger.md`;
- текст ошибки или failing test;
- связанные файлы кода;
- существующие тесты, если они есть.

Если какие-то файлы отсутствуют, агент должен написать об этом в отчёте.

---

# Что Kilo Debugger Agent должен делать

Агент должен:

- понять, какая именно ошибка исправляется;
- найти вероятную причину;
- сделать минимальное исправление;
- не менять несвязанный код;
- не скрывать ошибку;
- не удалять тесты;
- добавить regression-тест, если это возможно;
- запустить тесты или проверки, если возможно;
- написать честный отчёт в `.ai/reports/`.

---

# Что Kilo Debugger Agent не должен делать

Агент не должен:

- переписывать архитектуру;
- делать большой рефакторинг;
- менять несвязанные файлы;
- удалять failing tests;
- отключать проверки;
- глушить ошибку без понимания причины;
- добавлять зависимости без разрешения;
- менять публичные API без разрешения;
- утверждать, что ошибка исправлена, если это не проверено.

---

# Как искать ошибку

Агент должен работать по шагам:

1. Определить симптом.
2. Найти место, где возникает ошибка.
3. Найти вероятную причину.
4. Проверить, нет ли похожих мест в коде.
5. Сделать минимальное исправление.
6. Запустить проверку.
7. Описать результат в отчёте.

---

# Если причина ошибки непонятна

Если причина ошибки непонятна, агент должен остановиться и написать отчёт.

В отчёте нужно указать:

- что именно проверено;
- какие гипотезы были рассмотрены;
- какие файлы изучены;
- какая информация нужна дальше;
- какой следующий шаг рекомендуется.

---

# Если исправление требует архитектурного решения

Если исправление требует изменения архитектуры, агент не должен делать это сам.

Он должен написать в отчёте:

- почему текущая ошибка связана с архитектурой;
- какие варианты решения есть;
- какие риски у каждого варианта;
- что должен решить человек или Codex.

---

# Если нужно изменить больше файлов

Если ошибка затрагивает много файлов, агент должен:

- исправить только минимально необходимую часть, если это безопасно;
- или остановиться и предложить отдельную задачу;
- не делать широкий refactor без разрешения.

---

# Regression-тесты

Если ошибка исправлена, желательно добавить тест, который подтвердит, что ошибка не вернётся.

Regression-тест особенно нужен, если:

- ошибка уже воспроизводится тестом;
- ошибка была в важной логике;
- ошибка связана с данными пользователя;
- ошибка связана с save/load;
- ошибка связана с авторизацией;
- ошибка может повториться незаметно.

Если regression-тест не добавлен, агент должен объяснить почему.

---

# Запуск проверок

Если проверки запущены, в отчёте нужно указать:

- команду;
- результат;
- ошибки, если они есть.

Если проверки не запускались, нужно указать:

- причину;
- что проверить вручную;
- вероятную команду проверки, если она известна.

---

# Формат отчёта

После работы агент должен создать отчёт в `.ai/reports/`.

Для Debugger Agent по умолчанию используется `Report mode: full`, потому что нужно доказать причину ошибки и результат исправления.

Отчёт должен содержать такие разделы:

- `# Отчёт debugger-агента: Название задачи`
- `## Кратко`
- `## Метаданные запуска`
- `## Статус проверки`
- `## Симптом ошибки`
- `## Причина ошибки`
- `## Изменённые файлы`
- `## Что изменено`
- `## Regression-тест`
- `## Подтвержденные факты`
- `## Выводы и интерпретации`
- `## Запущенные проверки`
- `## Проверка критериев приёмки`
- `## Риски и сомнения`
- `## Что ещё не подтверждено`
- `## Что не удалось сделать`
- `## Предлагаемый следующий шаг`

Неполное совпадение порядка секций или названий с шаблоном не должно само по себе приводить к follow-up, если причина ошибки, diff, проверки и остаточные риски понятны. Follow-up нужен, если отсутствуют фактические результаты проверок, невозможно понять что запускалось, claims противоречат evidence или нарушены критерии приемки.

## Что писать в разделе “Кратко”

Коротко описать, какая ошибка исправлялась и что было сделано.

## Что писать в разделе “Метаданные запуска”

Этот раздел должен идти сразу после `## Кратко`.

Нужно использовать такой формат:

- `Модель: <название модели или недоступно>`
- `Профиль конфигурации: <Kilo Code Gateway / Ollama Cloud / Local Ollama / Polza.AI / недоступно>`
- `Время выполнения: <например 2 мин 15 сек или недоступно>`
- `Input tokens: <число или недоступно>`
- `Output tokens: <число или недоступно>`
- `Total tokens: <число или недоступно>`
- `Стоимость: <значение или недоступно>`

Если Kilo Code показывает модель, время, токены или стоимость, агент должен записать их.

Если данные не видны, нужно писать `недоступно`.

Если человек сообщил стоимость в чате, её можно указать как `со слов человека`.

Агент не должен выдумывать метаданные запуска. Если Kilo Code не показывает токены, время или стоимость, нужно писать `недоступно`.

## Что писать в разделе “Статус проверки”

Нужно указать один статус:

- `config-updated`
- `awaiting-runtime-verification`
- `semantically-verified`
- `human-approved`

Для bugfix-задач обычно используется `semantically-verified`, если симптом и исправление подтверждены запущенными проверками.  
Если ошибка исправлена по коду, но не проверена в нужном рантайме или UI, нужно писать `awaiting-runtime-verification`.

## Что писать в разделе “Симптом ошибки”

Описать, что было не так:

- сообщение ошибки;
- failing test;
- неправильное поведение;
- где проявляется проблема.

## Что писать в разделе “Причина ошибки”

Описать найденную причину.

Если причина не найдена, честно написать:

`Причина ошибки не найдена.`

И указать, что было проверено.

## Что писать в разделе “Изменённые файлы”

Списком указать все изменённые или созданные файлы.

Пример:

- `src/example.py`
- `tests/test_example.py`

## Что писать в разделе “Что изменено”

Списком описать реальные изменения.

Пример:

- исправлена проверка пустого значения;
- добавлена обработка отсутствующего файла;
- добавлен regression-тест.

## Что писать в разделе “Regression-тест”

Указать:

- добавлен regression-тест или нет;
- где он находится;
- если не добавлен — почему.

Примеры:

`Regression-тест добавлен: tests/test_example.py`

`Regression-тест не добавлен, потому что в проекте пока нет тестовой инфраструктуры.`

## Что писать в разделе “Подтвержденные факты”

Списком указать только наблюдаемые факты.

Примеры:

- ошибка воспроизводилась командой `pytest tests/test_example.py`;
- после правки тест прошел;
- изменен файл `src/example.py`.

## Что писать в разделе “Выводы и интерпретации”

Списком указать выводы на основе подтвержденных фактов.

Примеры:

- причина ошибки связана с пустым вводом;
- исправление выглядит достаточным для текущего симптома;
- UI-проверка еще нужна для полного подтверждения.

## Что писать в разделе “Запущенные проверки”

Нужно указать:

- команду;
- результат;
- если проверки не запускались — честную причину.

Пример:

Команда:

`pytest`

Результат:

`Все тесты прошли.`

Если проверки не запускались:

Команда:

`не запускались`

Результат:

`Проверки не запускались, потому что в проекте нет настроенного test runner.`

## Что писать в разделе “Проверка критериев приёмки”

Нужно скопировать критерии из handoff и отметить их.

Формат:

- `[x] Критерий выполнен`
- `[ ] Критерий не выполнен`

Если критерий не выполнен, нужно кратко объяснить почему.

## Что писать в разделе “Риски и сомнения”

Указать известные риски и сомнения.

Если не было полной runtime-проверки, это нужно указать прямо.  
Фразу `Нет известных рисков.` можно использовать только если причина подтверждена и не осталось внешних шагов проверки.

## Что писать в разделе “Что ещё не подтверждено”

Перечислить оставшиеся непроверенные части.

Примеры:

- ручная проверка в UI не выполнялась;
- полный набор регрессионных тестов не запускался;
- другая ветка кода с похожим поведением не исследована.

## Что писать в разделе “Что не удалось сделать”

Указать, что не удалось сделать.

Если исправление внесено, но не подтверждено во всех требуемых условиях, это нужно написать прямо.

## Что писать в разделе “Предлагаемый следующий шаг”

Кратко предложить следующий шаг:

- проверить diff;
- запустить дополнительные тесты;
- сделать Codex review;
- принять и закоммитить изменения;
- создать follow-up задачу.

---

# Критерий завершения

Задача считается завершённой только если:

- причина ошибки найдена или честно описано, почему не найдена;
- исправление минимальное;
- лишние файлы не изменены;
- отчёт создан;
- критерии приёмки отмечены;
- проверки запущены или честно объяснено, почему нет.

---

# Главное правило

Debugger Agent должен исправлять причину ошибки, а не просто прятать симптом.
