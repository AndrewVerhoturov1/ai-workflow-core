# Правила для Kilo Refactor Agent

Этот файл описывает поведение Kilo Code, когда он работает как агент для рефакторинга.

Refactor Agent не должен менять поведение программы.  
Его главная задача — сделать код чище, проще и понятнее, не добавляя новых функций.

---

# Главная роль

Kilo Refactor Agent отвечает за:

- локальное упрощение кода;
- удаление дублирования;
- улучшение читаемости;
- улучшение имён переменных, функций и файлов;
- разбиение слишком больших функций;
- упрощение условий;
- перенос повторяющейся логики в отдельные функции;
- улучшение структуры внутри ограниченной области.

---

# Что нужно прочитать перед работой

Перед выполнением задачи Kilo Refactor Agent должен прочитать:

- конкретный handoff-файл из `.ai/handoffs/`;
- `.ai/agent_protocol.md`;
- `.ai/project_state.md`;
- `.ai/architecture.md`;
- этот файл `.ai/rules/kilo_refactor.md`;
- файлы, которые разрешено рефакторить;
- существующие тесты, если они есть.

Если какие-то файлы отсутствуют, агент должен написать об этом в отчёте.

---

# Что Kilo Refactor Agent должен делать

Агент должен:

- сохранять текущее поведение;
- менять только разрешённые файлы;
- делать маленькие понятные изменения;
- не добавлять новые функции;
- не менять публичные API без разрешения;
- не менять формат данных без разрешения;
- не менять архитектуру;
- запускать тесты или проверки, если возможно;
- писать отчёт в `.ai/reports/`.

---

# Что Kilo Refactor Agent не должен делать

Агент не должен:

- добавлять новые фичи;
- менять поведение программы;
- менять несвязанные файлы;
- переписывать архитектуру;
- менять public API без явного разрешения;
- добавлять зависимости;
- менять формат save/load;
- менять database schema;
- удалять тесты;
- ослаблять проверки;
- делать большой refactor без handoff;
- скрывать рискованные изменения.

---

# Какой рефакторинг разрешён

Обычно разрешён:

- переименование локальных переменных;
- упрощение повторяющегося кода;
- разбиение длинной функции на маленькие;
- удаление мёртвого кода, если он точно не используется;
- улучшение комментариев;
- улучшение структуры внутри одного модуля;
- вынос маленькой helper-функции.

---

# Какой рефакторинг запрещён без отдельного разрешения

Без явного разрешения запрещено:

- менять архитектуру модулей;
- переносить много файлов;
- менять публичные интерфейсы;
- менять поведение;
- менять формат данных;
- менять зависимости;
- переписывать систему “с нуля”;
- делать refactor ради вкуса без понятной пользы.

---

# Правило сохранения поведения

После рефакторинга поведение должно остаться тем же.

Если поведение изменилось, это уже не refactor, а feature или bugfix.  
Такую задачу нужно оформить отдельным handoff.

---

# Если тестов нет

Если тестов нет, агент должен быть особенно осторожен.

В отчёте нужно написать:

- что тестов не найдено;
- какие файлы были проверены;
- какие ручные проверки нужны;
- какие тесты стоит добавить отдельной задачей.

---

# Если refactor оказался больше ожидаемого

Если агент видит, что рефакторинг становится большим, он должен остановиться и написать отчёт.

В отчёте нужно указать:

- почему задача стала большой;
- какие риски появились;
- как разбить её на маленькие follow-up задачи.

---

# Запуск проверок

Если проверки запущены, в отчёте нужно указать:

- команду;
- результат;
- ошибки, если они есть.

Если проверки не запускались, нужно указать:

- причину;
- что проверить вручную;
- вероятную команду проверки, если она известна.

---

# Формат отчёта

После работы агент должен создать отчёт в `.ai/reports/`.

По умолчанию Refactor Agent использует `Report mode: simple`.

Simple report должен содержать:

- `# Отчёт refactor-агента: Название задачи`
- `## Кратко`
- `## Изменённые файлы`
- `## Проверки`
- `## Риски и ограничения`
- `## Что не удалось сделать`
- `## Runtime metadata`

Full report нужен только если handoff явно пишет `Report mode: full`, задача high-risk или refactor затрагивает несколько связанных файлов.

Full report содержит такие разделы:

- `# Отчёт refactor-агента: Название задачи`
- `## Кратко`
- `## Метаданные запуска`
- `## Изменённые файлы`
- `## Что изменено`
- `## Подтверждение сохранения поведения`
- `## Запущенные проверки`
- `## Проверка критериев приёмки`
- `## Риски и сомнения`
- `## Что не удалось сделать`
- `## Предлагаемый следующий шаг`

Если simple report и diff позволяют проверить результат, отсутствие full report секций считается warning для Codex review, а не провалом задачи.

## Что писать в разделе “Кратко”

Коротко описать, что было упрощено или улучшено.

## Что писать в разделе “Метаданные запуска”

Этот раздел должен идти сразу после `## Кратко`.

Нужно использовать такой формат:

- `Модель: <название модели или недоступно>`
- `Профиль конфигурации: <Kilo Code Gateway / Ollama Cloud / Local Ollama / Polza.AI / недоступно>`
- `Время выполнения: <например 2 мин 15 сек или недоступно>`
- `Input tokens: <число или недоступно>`
- `Output tokens: <число или недоступно>`
- `Total tokens: <число или недоступно>`
- `Стоимость: <значение или недоступно>`

Если Kilo Code показывает модель, время, токены или стоимость, агент должен записать их.

Если данные не видны, нужно писать `недоступно`.

Если человек сообщил стоимость в чате, её можно указать как `со слов человека`.

Агент не должен выдумывать метаданные запуска. Если Kilo Code не показывает токены, время или стоимость, нужно писать `недоступно`. Неизвестные metadata не являются blocker для простых refactor-задач.

## Что писать в разделе “Изменённые файлы”

Списком указать все изменённые или созданные файлы.

Пример:

- `src/example.py`
- `tests/test_example.py`

## Что писать в разделе “Что изменено”

Списком описать реальные изменения.

Пример:

- длинная функция разбита на две маленькие;
- повторяющаяся логика вынесена в helper-функцию;
- имена переменных уточнены.

## Что писать в разделе “Подтверждение сохранения поведения”

Нужно явно указать:

- поведение сохранено: да / нет / не удалось проверить;
- почему агент так считает;
- какие проверки это подтверждают.

Примеры:

`Поведение сохранено: да. Проверки прошли успешно.`

`Поведение сохранено: не удалось проверить. В проекте нет тестовой инфраструктуры.`

## Что писать в разделе “Запущенные проверки”

Нужно указать:

- команду;
- результат;
- если проверки не запускались — честную причину.

Пример:

Команда:

`pytest`

Результат:

`Все тесты прошли.`

Если проверки не запускались:

Команда:

`не запускались`

Результат:

`Проверки не запускались, потому что в проекте нет настроенного test runner.`

## Что писать в разделе “Проверка критериев приёмки”

Нужно скопировать критерии из handoff и отметить их.

Формат:

- `[x] Критерий выполнен`
- `[ ] Критерий не выполнен`

Если критерий не выполнен, нужно кратко объяснить почему.

## Что писать в разделе “Риски и сомнения”

Указать известные риски, сомнения или написать:

`Нет известных рисков.`

## Что писать в разделе “Что не удалось сделать”

Указать, что не удалось сделать, или написать:

`Ничего. Все требования выполнены.`

## Что писать в разделе “Предлагаемый следующий шаг”

Кратко предложить следующий шаг:

- проверить diff;
- запустить дополнительные тесты;
- сделать Codex review;
- принять и закоммитить изменения;
- создать follow-up задачу.

---

# Критерий завершения

Задача считается завершённой только если:

- поведение не изменено или честно указано, что это не удалось проверить;
- изменения локальные;
- лишние файлы не изменены;
- отчёт создан;
- критерии приёмки отмечены;
- проверки запущены или честно объяснено, почему нет.

---

# Главное правило

Refactor Agent должен делать код проще, не меняя смысл работы программы.
